# v2.4.0 Phase 3: Wave-Based Remote Staging

## Current state

Phase 1 (queue size columns) and Phase 2 (AV1 support, per-file codec resolution, code quality pass) are complete and shipped as v2.4.1. The codebase already has:

- **Per-file staging** in `cli_main.rs` - stages each remote file individually before encoding, copies output back for replace mode on remote mounts. This works but is inefficient: each file is staged, encoded, cleaned up, then the next file is staged. On a batch of 50 remote files, that's 50 sequential copy-encode-cleanup cycles with no overlap or batching.
- **`StagingContext`** in `staging.rs` - handles copy-in, path rewriting, and Drop-based cleanup for a single file. Has free-space checks.
- **`MountCache`** in `remote.rs` - detects remote mounts on all platforms with directory-level canonicalise caching.
- **`DiskMonitor`** in `disk_monitor.rs` - runtime partition monitoring with async wait-for-space.
- **`BatchSettings.force_local: bool`** - already on the struct, always set to `false`.
- **`--remote auto|always|never`** CLI flag - already parsed and used.
- **`FfmpegProgress`** in `encoder.rs` - shared progress parser for stderr.

## Goal

Replace per-file staging with wave-based staging. The wave planner groups remote files into waves that fit in available staging space, stages an entire wave at once, encodes the wave, cleans up, and repeats. Local files are encoded in-place between waves.

## Architecture

### New: `staging::WavePlanner`

Partitions the pending queue into an ordered sequence of `WaveItem`s:

```rust
pub enum WaveItem {
    /// Encode this local file directly (no staging needed).
    Local { queue_index: usize },
    /// A wave of remote files to stage together.
    Wave {
        indices: Vec<usize>,
        total_stage_bytes: u64,
    },
}
```

The planner walks the queue in order. Consecutive remote files accumulate into a wave until adding the next file would exceed `available_space * 0.9`. At that point the wave is flushed and a new one starts. Local files between remote files flush any pending wave and insert themselves as `WaveItem::Local`.

The planner does not stage files itself - it only decides the grouping. The encode loop consumes wave items and orchestrates staging/cleanup.

### Modified: `run_encode_loop` (encoder.rs)

Currently iterates `pending_indices` directly. Phase 3 changes the outer loop to consume `WaveItem`s from the planner:

```
for wave_item in plan:
    match wave_item:
        Local { idx } -> encode file at idx (unchanged from current behaviour)
        Wave { indices, .. } ->
            stage all files in the wave (with aggregate progress)
            for idx in indices:
                encode file at idx (using staged local path)
            cleanup all staged files in the wave
```

The inner per-file encode logic (decision, ffmpeg spawn, post-encode checks) is unchanged. Only the outer loop and path resolution change.

### Modified: `EventSink` (events.rs)

Four new methods:

```rust
fn wave_progress(&self, wave: u32, total_waves: u32, file_in_wave: u32, wave_size: u32);
fn wave_status(&self, message: &str);
fn batch_time_estimate(&self, elapsed_secs: f64, remaining_secs: f64);
fn wave_time_estimate(&self, elapsed_secs: f64, remaining_secs: f64);
```

All existing implementations (TauriSink, CliSink) get default no-op implementations initially to avoid breaking the build. Wire up actual display in subsequent steps.

### Modified: CLI (`cli_main.rs`)

The existing per-file staging loop (the `should_stage` block that iterates `queue_items` and creates individual `StagingContext`s) is removed. Wave planning and staging moves into `run_encode_loop` so both GUI and CLI get it automatically.

### GUI additions

- New checkbox: "Transcode over network" in the Behaviour section. Tooltip: "Skip local staging and read/write directly over the network. Fast on a wired LAN, slow or unreliable over Wi-Fi or WAN."
- Wired to `BatchSettings.force_local` (inverted: checked = `force_local: true`, meaning don't stage).
- Status strip shows wave context when encoding remote files: "Wave 2/4 [3/8 files]".
- Config save/load for the new checkbox.

## Implementation steps

### 3.1 EventSink additions

Add the four new methods to the `EventSink` trait with default no-op implementations. Add empty implementations to `TauriSink` and `CliSink`.

**Files:** `events.rs`, `tauri_sink.rs`, `cli_sink.rs`

**Verification:** `cargo build` for both GUI and CLI targets.

### 3.2 WavePlanner

Create `staging::WavePlanner` and `staging::WaveItem`. The planner takes:
- A slice of `&QueueItem` (the pending items)
- A `&MountCache` (for remote detection)
- The staging directory path
- The remote policy (auto/always/never)

Returns `Vec<WaveItem>`.

The planner queries `partition_free_space` on the staging directory once at construction and uses 90% of available space as the wave budget. Files whose `source_bytes` alone exceed the budget get their own single-file wave (they have to be staged individually regardless).

For `--remote never` or when all files are local, the planner returns one `WaveItem::Local` per file with no waves.

**Files:** `staging.rs`

**Verification:** Unit test with mock queue items and known sizes, asserting correct wave grouping.

### 3.3 Wave-aware encode loop

Modify `run_encode_loop` in `encoder.rs` to accept an optional `WavePlan` (the planner output). When present, the loop consumes wave items instead of iterating `pending_indices` directly.

For `WaveItem::Wave`, the loop:
1. Calls `sink.wave_status(...)` with "Staging wave N/M (X files, Y MB)"
2. Stages all files in the wave using `StagingContext::stage_file` for each
3. Rewrites `queue[idx].full_path` to the staged local path
4. Encodes each file in the wave using the existing per-file logic
5. After all files in the wave are encoded (or cancelled), cleans up all staging contexts
6. Restores `queue[idx].full_path` to the original remote path (needed for replace mode copy-back and for the log to show the real path)

For `WaveItem::Local`, the loop encodes directly as before.

The `WavePlan` is `Option<Vec<WaveItem>>` - when `None`, the loop falls back to the current flat iteration over `pending_indices`. This preserves backward compatibility for the GUI until it's wired up.

**Files:** `encoder.rs`

**Verification:** CLI dry-run with `--remote auto` on a mixed local/remote queue. Confirm wave grouping in the plan output.

### 3.4 CLI integration

Remove the existing per-file staging block in `cli_main.rs` (the `should_stage` / `staging_contexts` / `original_remote_paths` section). Instead:
1. Build a `WavePlanner` before calling `run_encode_loop`
2. Pass the plan to `run_encode_loop`
3. Remove the post-loop remote copy-back logic (this is now handled inside the wave-aware loop)

The `MountCache` instance already exists at this point in `cli_main.rs`.

**Files:** `cli_main.rs`

**Verification:** Encode a batch containing both local and remote files. Confirm staging happens per-wave, not per-file. Confirm replace mode on remote mounts works (output copied back to remote).

### 3.5 CLI progress display

Wire the new `EventSink` methods in `CliSink`:
- `wave_status` prints the wave header line
- `wave_progress` updates the TTY progress display with wave context
- `batch_time_estimate` / `wave_time_estimate` print ETA when available

Time estimates use a rolling average of seconds-per-file. The wave estimate uses only files in the current wave; the batch estimate uses all completed files. Display "calculating..." until at least one file has completed.

**Files:** `cli_sink.rs`

**Verification:** Encode a remote batch and confirm the wave context appears in the progress output.

### 3.6 GUI: "Transcode over network" checkbox

Add the checkbox to `index.html` in the Behaviour section (after the existing checkboxes, before Performance). Wire to:
- `BatchSettings.force_local` in the `start_batch` command (lib.rs)
- Config save/load (config.rs)
- When checked, pass `force_local: true` so the planner skips staging

**Files:** `index.html`, `lib.rs`, `config.rs`

**Verification:** Check the box, start a batch with remote files. Confirm files are encoded over the network without staging.

### 3.7 GUI: wave progress display

Wire the new `EventSink` methods in `TauriSink`:
- Emit `wave-status` and `wave-progress` events

```
Wave 2/4 [3/8 files] - Encoding file 15 of 42
```

The batch progress bar interpolates across waves (completed waves contribute their full weight, the current wave contributes proportionally).

**Files:** `tauri_sink.rs`, `index.html`

## Risk assessment

The highest-risk step is 3.3 (wave-aware encode loop). The encode loop is already ~900 lines and this adds another layer of nesting. Mitigations:
- The wave plan is consumed as a simple iterator; the inner per-file logic is unchanged
- `WavePlan` is `Option` so the loop degrades gracefully to flat iteration
- Each wave's staging contexts are collected in a Vec and cleaned up in a `Drop` guard so Ctrl+C during encoding still removes staged files
- Build and test after each step before proceeding

Steps 3.1-3.2 are zero-risk (additive types, no existing code modified). Steps 3.4-3.8 are medium-risk (wiring changes with testable outputs). Step 3.3 should be implemented and tested in isolation before 3.4 wires it into the CLI.
